## Context

### The Domain Model First

Before any architectural decision was made, the team conducted a **Domain-Driven Design (DDD) event storming session** to map out the RuralGig business domain. The output of that session — not a technical requirement, but the **domain model itself** — drove this decision.

Event storming revealed the following **Bounded Contexts** as natural seams in the business:

| Bounded Context | Core Responsibility | Aggregate Root |
|---|---|---|
| **Task Context** | The lifecycle of a task from posting to payment | `Task` |
| **Worker Context** | A worker's identity, skills, and earnings | `Worker` |
| **Matching Context** | The act of connecting a worker to a task | `MatchResult` |
| **Payment Context** | Money movement and wallet state | `Wallet` |
| **Notification Context** | Delivery of messages to users | `Notification` |
| **Identity Context** | Authentication and authorisation | `User` |

Each bounded context has a **clear linguistic boundary** — a Ubiquitous Language specific to that context. The word "Task" means something precise and complete in the Task Context. It does not bleed into the Payment Context, where the relevant concept is a `PayoutInstruction`, not a `Task`. This linguistic separation is not cosmetic. It reflects real business rules that must not be entangled in code.

**This DDD analysis directly determined our service boundaries.** Each NestJS service maps to exactly one bounded context. The architectural question was then: how should these contexts communicate?

---

### The Two Workflows That Forced the Decision

The event storming session identified two critical cross-context workflows:

**Task Lifecycle Workflow** — spans Task, Matching, and Notification contexts:
```
[Task Context]         Task is claimed by a worker
                            │
                            │ crosses context boundary
                            ▼
[Matching Context]     Remove task from available pool
[Notification Context] Send claim confirmation to worker and business
```

**Payment Workflow** — spans Task, Payment, Worker, and Notification contexts:
```
[Task Context]         Business approves submitted work
                            │
                            │ crosses context boundary
                            ▼
[Payment Context]      Initiate bKash payout
[Worker Context]       Update earnings ledger
[Notification Context] Send payment confirmation SMS
```

The critical DDD insight here is that **each context owns its own reaction** to what happened. The Task Context does not need to know that the Matching Context exists, or that workers receive SMS notifications. In DDD terms, the `Task` aggregate does not hold a reference to a `MatchResult` aggregate or a `Notification` aggregate. Cross-aggregate, cross-context communication must be indirect — and domain events are the DDD-prescribed mechanism for exactly this.

---

## Options Considered

### Option A: Synchronous REST (Service-to-Service HTTP Calls)

Each service directly calls the downstream services over HTTP when a state change occurs.

**How it would look for task claiming:**

```
Worker calls POST /tasks/:id/claim
        │
        ▼
  task-service                        ← Task Context
  (updates DB)
        │
        ├──► HTTP POST matching-service/tasks/:id/remove-from-pool
        │    (task-service reaches INTO the Matching Context)
        │    (waits for 200 OK)
        │
        ├──► HTTP POST notification-service/sms/send
        │    (task-service reaches INTO the Notification Context)
        │    (waits for 200 OK)
        │
        └──► HTTP POST notification-service/push/send
             (waits for 200 OK)
        │
  returns 200 to worker
```

**DDD problem with this approach:** `task-service` is now directly coupled to the internal APIs of both the Matching Context and Notification Context. It holds knowledge of how those contexts work internally. This is a fundamental DDD anti-pattern called **context leakage** — the concerns of one bounded context bleed into another.

The `Task` aggregate root is responsible for enforcing task business rules. It should not orchestrate SMS delivery. When the Notification Context changes how it sends SMS (for example, switching from Twilio to a local Bangladeshi provider), `task-service` may need to change too. The contexts are no longer independent. Their teams cannot work autonomously. Their deployments are coupled.

### Option B: Asynchronous Event-Driven (RabbitMQ Domain Events)

When a state change occurs inside a bounded context, the **aggregate root raises a domain event**. The originating service publishes that event to RabbitMQ. All other bounded contexts that care about this event subscribe and react on their own terms, through their own Anti-Corruption Layers.

**How it looks for task claiming:**

```
Worker calls POST /tasks/:id/claim
        │
        ▼
  task-service                        ← Task Context
  Task aggregate enforces invariants:
    - is task status OPEN?
    - has deadline not passed?
    - acquire Redlock mutex (prevent double-claim)
  Task aggregate raises: TaskClaimedEvent  ← pure domain event
  (published to RabbitMQ exchange)
        │
  returns 200 to worker immediately
        │
  RabbitMQ delivers TaskClaimedEvent to subscribing contexts:
        ├──► matching-service    ← Matching Context reacts autonomously via its own ACL
        ├──► notification-service ← Notification Context reacts autonomously via its own ACL
        └──► (future contexts subscribe here — zero changes to task-service)
```

**DDD alignment:** The `Task` aggregate root does exactly what DDD prescribes — it enforces invariants, changes state, and raises a domain event. It has no knowledge of what other contexts do with that event. Each subscribing context translates the event through its own **Anti-Corruption Layer (ACL)**, converts it into its own domain language, and applies its own business rules independently.

---

## Decision

**We chose Option B: asynchronous event-driven architecture using RabbitMQ.**

This decision was driven primarily by **DDD principles**, with technical performance and resilience benefits as supporting evidence.

---

## Reasoning

### 1. DDD Bounded Context Autonomy (Primary Reason)

The most important reason is structural to DDD itself. A bounded context is defined as a self-contained model with its own language, rules, and data. **A bounded context that can only function when another bounded context is available is not truly bounded.**

With synchronous REST, the Task Context has a runtime dependency on the Matching Context and Notification Context. If either goes down, the Task Context cannot complete a task claim — even though the task claiming logic itself is entirely valid and all domain invariants are satisfied. The `Task` aggregate's rules are sound; it is the infrastructure coupling that introduces the failure.

With domain events, each bounded context is truly autonomous at runtime. The Task Context can enforce its rules and publish its events whether or not the Matching or Notification contexts are running. This is bounded context autonomy as DDD intends it.

```
DDD principle:
  Each bounded context owns its own data, its own rules,
  and its own reactions to events from other contexts.

Synchronous REST violates this:
  task-service "owns" the notification logic by deciding
  when and how to call notification-service.

Domain events respect this:
  notification-service decides when and how to react to
  TaskClaimedEvent entirely on its own terms.
```

### 2. Aggregate Root Raises Domain Events — the DDD Pattern

In DDD, an **aggregate root** is the single entry point for all state changes within an aggregate cluster. After enforcing invariants and persisting the change, the aggregate root **raises a domain event** — a pure domain notification, not an infrastructure call, that declares something meaningful happened in the business.

```typescript
// task-service/src/domain/task.entity.ts
export class Task extends AggregateRoot {
  claim(workerId: string): void {
    // ── Enforce aggregate invariants — pure domain logic, no infrastructure ──
    if (this.status !== TaskStatus.OPEN) {
      throw new TaskNotClaimableError(this.id, this.status);
    }
    if (this.deadline < new Date()) {
      throw new TaskDeadlinePassedError(this.id);
    }

    // ── Change aggregate state ────────────────────────────────────────────────
    this.status = TaskStatus.CLAIMED;
    this.claimedByWorkerId = workerId;
    this.claimedAt = new Date();

    // ── Raise domain event ────────────────────────────────────────────────────
    // The aggregate ANNOUNCES what happened in the domain.
    // It does NOT call notification-service.
    // It does NOT call matching-service.
    // It declares: "a task was claimed." Other contexts decide what to do.
    this.apply(new TaskClaimedEvent({
      taskId:         this.id,
      workerId,
      claimedAt:      this.claimedAt,
      requiredSkills: this.requiredSkills,
      budget:         this.budget,
    }));
  }
}
```

This is DDD as Eric Evans describes it in the blue book. The aggregate root is responsible for business rules only. What to do next — notify a worker, remove from a pool, update analytics — belongs to other contexts that react to the event through their own domain logic.

### 3. Ubiquitous Language Preserved Across Context Boundaries via Anti-Corruption Layers

Each bounded context has its own Ubiquitous Language. A core DDD challenge is preventing one context's language from contaminating another's. Event-driven communication enforces clean boundaries through two mechanisms:

**First**, the domain event payload uses only the publishing context's language. When `task-service` publishes `TaskClaimedEvent`, the payload contains Task Context terms: `taskId`, `workerId`, `claimedAt`, `requiredSkills`. It does not contain Matching Context language (`matchScore`, `poolPosition`) or Notification Context language (`smsTemplate`, `recipientPhone`).

**Second**, each subscribing context translates the event through its own **Anti-Corruption Layer (ACL)** — a dedicated translator class that converts the incoming event into that context's own domain model and language:

```typescript
// matching-service/src/acl/task-events.translator.ts
// Anti-Corruption Layer — Matching Context's translation of Task Context events
export class TaskEventsTranslator {
  fromTaskClaimedEvent(event: TaskClaimedEvent): RemoveFromPoolCommand {
    // The Matching Context does not care about workerId or budget.
    // It translates the event into its own domain command using its own language.
    return new RemoveFromPoolCommand({
      taskId:        event.payload.taskId,
      removalReason: PoolRemovalReason.CLAIMED,  // Matching Context enum — not in Task Context
    });
  }
}

// notification-service/src/acl/task-events.translator.ts
// Anti-Corruption Layer — Notification Context's translation of Task Context events
export class TaskEventsTranslator {
  fromTaskClaimedEvent(event: TaskClaimedEvent): SendNotificationCommand {
    // The Notification Context translates taskId + workerId into
    // its own domain language: recipient, channel, template.
    return new SendNotificationCommand({
      recipientId: event.payload.workerId,
      channel:     NotificationChannel.SMS,      // Notification Context enum
      template:    SmsTemplate.TASK_CLAIM_CONFIRM, // Notification Context concept
      variables:   { taskId: event.payload.taskId },
    });
  }
}
```

With synchronous REST, `task-service` would need to construct both the Matching Context's HTTP payload and the Notification Context's HTTP payload — contaminating the Task Context's codebase with the language of other contexts. The ACL pattern, enabled by domain events, keeps each context's language clean and isolated.

### 4. Domain Events as the Authoritative Cross-Context Record

In DDD, cross-context consistency is achieved through domain events, not distributed transactions. The RabbitMQ event stream becomes the **authoritative record** of everything that happened in the business domain — more durable and more semantically meaningful than HTTP access logs.

Each domain event is immutable, timestamped, and named after a past-tense business fact. The sequence of events for a single task — `TaskCreatedEvent` → `TaskClaimedEvent` → `TaskSubmittedEvent` → `TaskApprovedEvent` → `TaskPaidEvent` — is the complete, human-readable story of that task's life in the business.

This event stream is the foundation for multiple downstream capabilities in RuralGig:

- **CQRS read models** (ADR-002) — the task feed read model is built by projecting `TaskCreatedEvent` and `TaskClaimedEvent` into a Redis cache. The read model is not a separate database query — it is a projection of domain history.
- **Audit trails** — regulators or businesses asking "what happened to task X?" get a complete, immutable answer from the event log.
- **Event replay** — if the Matching Context's pool gets corrupted, it can be fully rebuilt by replaying all historical `TaskCreatedEvent` and `TaskClaimedEvent` events from the beginning of time.
- **Analytics** — business metrics (task claim rate, average time to claim, worker retention) are computed from the event stream without touching operational databases.

### 5. Failure Isolation Aligned with DDD Context Boundaries

In DDD, each bounded context has its own failure modes. A failure in the Notification Context — an SMS gateway timeout at 2am — is a failure in that context alone. It should not propagate into the Task Context and prevent a worker from claiming a task.

With synchronous REST, a crash in `notification-service` causes `task-service` to return a 500 error to the worker. From a domain perspective, nothing is wrong. The task is claimable. The worker is eligible. Every invariant is satisfied. The failure is entirely in a secondary infrastructure concern that has no bearing on the core domain action.

With domain events, the Task Context enforces its invariants successfully and publishes the event. The worker gets a 200 response. RabbitMQ holds the `TaskClaimedEvent` in the queue. When `notification-service` recovers, it consumes the event and delivers the SMS — delayed by minutes, never lost. **The domain is sound; only the infrastructure had a hiccup.** This is DDD's bounded context isolation expressed as a runtime property.

### 6. Open/Closed Principle at the Architecture Level

DDD's strategic design encourages bounded contexts that evolve independently. Domain events achieve this by applying the Open/Closed Principle architecturally: the system is open for extension (new contexts subscribe to existing events) and closed for modification (existing contexts do not change when new subscribers are added).

```
Today — subscribers to TaskClaimedEvent:
  matching-service      (removes task from available pool)
  notification-service  (sends SMS confirmation)

In 3 months — new subscribers, zero changes to task-service:
  analytics-service     (tracks claim velocity by skill category)
  fraud-service         (flags suspiciously fast claims for review)
  search-service        (updates task search index)
  ai-service            (triggers career suggestion if worker's 5th claim)
```

In a synchronous REST architecture, each of these additions requires modifying `task-service`, writing new HTTP client code, adding new configuration, retesting the claim endpoint, and redeploying. With domain events, each new bounded context is independently developed and deployed, subscribing to the exchange without any coordination with the Task Context team.

### 7. Performance (Supporting Technical Reason)

This is a secondary reason, not the primary driver. With synchronous REST, the claim endpoint is as slow as its slowest downstream call:

```
task-service (5ms DB write)
  → matching-service (20ms pool update)
  → notification-service (150ms — waits for SMS gateway response)
= 175ms minimum for the worker to get a response
```

With domain events, the worker waits only for the Task Context to enforce its rules and persist its state: **~5ms**. The rest happens asynchronously. On a mobile connection in rural Bangladesh where network latency is already significant, this 35x improvement in response time is meaningful for user experience and task claim conversion rate.

## Trade-offs and Mitigations

### Trade-off 1: Eventual Consistency Across Bounded Contexts

**The problem:** When a task is claimed, the Matching Context does not remove it from its pool instantly. For a brief window of milliseconds to seconds, another worker browsing the task feed may see it as available. If they attempt to claim it, they receive a "task already claimed" error.

**DDD perspective — this is expected and correct behaviour:** DDD explicitly teaches that **cross-context consistency is eventual, not immediate.** The Task Context is the authoritative source of truth for task state. Its data is always consistent — the task is CLAIMED the moment the aggregate root's `claim()` method succeeds. The Matching Context's available-tasks pool is a **read model projection** of Task Context events. Like all projections, it will catch up within seconds.

This is not a bug. It is the correct behaviour of a distributed domain model. Asking for immediate cross-context consistency would require a distributed transaction — which DDD and modern distributed systems design both reject as impractical at scale.

**Mitigation:** The Redis distributed Redlock mutex on `task-service` prevents double-claiming at the aggregate root level, regardless of what read model projections other contexts have processed. Invariants are enforced at the aggregate boundary — exactly where DDD says they must be enforced. No worker will ever be double-assigned to the same task.

### Trade-off 2: Distributed Tracing Requires Explicit Correlation

**The problem:** With synchronous REST, a request ID flows through the call chain automatically via HTTP headers. With domain events, each context processes the event in a separate process. There is no automatic link between the originating HTTP request and the downstream event processing.

**DDD perspective:** This is a natural consequence of context autonomy. Each context is a separate process with its own execution context. Tracing across them is an infrastructure concern that must be solved explicitly.

**Mitigation:** Every domain event carries structured DDD tracing fields:

```typescript
// packages/shared-events/src/base.event.ts
export interface DomainEvent {
  eventId:        string;  // unique ID for this event instance (UUID)
  eventType:      string;  // e.g. 'task.claimed' — past tense business fact
  occurredAt:     string;  // ISO timestamp — when it happened in the domain
  aggregateId:    string;  // ID of the aggregate that raised the event
  aggregateType:  string;  // e.g. 'Task' — which DDD aggregate owns this
  correlationId:  string;  // original HTTP request ID — for distributed tracing
  causationId:    string;  // ID of the command/event that caused this event
  boundedContext: string;  // e.g. 'task-context' — which context published this
  schemaVersion:  number;  // event schema version — for backward compatibility
}
```

The `causationId` field enables **event causality chains** — you can trace not just which HTTP request started a workflow, but which event caused which downstream event, enabling full DDD event sourcing replay and causal debugging.

### Trade-off 3: RabbitMQ Outage Stops Event Delivery

**The problem:** If RabbitMQ goes down in production, domain events are not delivered and downstream contexts stop reacting to domain changes.

**DDD mitigation — Outbox Pattern:** This is solved by a DDD-aligned pattern that ensures domain state and domain events are always consistent. Events are first persisted to an `outbox` table inside the same database transaction as the aggregate state change. A separate publisher process polls the outbox and sends events to RabbitMQ. If RabbitMQ is unavailable, events accumulate in the outbox and are published when it recovers.

```
Without Outbox (dangerous):
  BEGIN TRANSACTION
    UPDATE tasks SET status = 'CLAIMED', claimed_by = :workerId
  COMMIT
  ← RabbitMQ crashes here → TaskClaimedEvent is LOST
  ← Matching Context never removes the task from the pool
  ← Notification Context never sends the SMS
  ← Domain state and event log are now inconsistent

With Outbox Pattern (DDD-safe):
  BEGIN TRANSACTION
    UPDATE tasks SET status = 'CLAIMED', claimed_by = :workerId
    INSERT INTO outbox (event_type, payload, published) VALUES ('task.claimed', {...}, false)
  COMMIT   ← atomic: aggregate state + outbox entry saved together
  ← Separate poller reads unpublished outbox entries
  ← Publishes each event to RabbitMQ, marks as published
  ← If RabbitMQ is down, poller retries — no events are ever lost
```

This guarantees that if the `Task` aggregate successfully changed state, the `TaskClaimedEvent` will eventually be published — a DDD-level guarantee that domain state and domain events are never inconsistent.

### Trade-off 4: More Complex Local Development

**The problem:** Running eight services and RabbitMQ locally requires more setup than a monolith.

**Mitigation:** `docker compose up -d` from `infra/` starts PostgreSQL, Redis, and RabbitMQ in a single command. `npx turbo dev` starts all services in parallel. For focused development on a single bounded context, other services can be omitted — RabbitMQ buffers their events and delivers them when those services restart, which is exactly the resilience guarantee we want in production too.

## Consequences

### What becomes easier

- Any new bounded context can subscribe to existing domain events without modifying any existing service
- Each context can be developed, tested, and deployed by an independent team in complete isolation
- Task claiming responds in ~5ms regardless of how many contexts react to the event downstream
- A failure in the Notification Context has zero impact on the Task Context's ability to enforce domain rules
- The domain event log is a complete, immutable, human-readable audit trail of every business fact — queryable for analytics, replayable for debugging and rebuilding read models
- Future requirements like event sourcing, time-travel debugging, and CQRS read model rebuilding are structurally possible because the event history is preserved

### What becomes harder

- The system is eventually consistent across bounded context boundaries — every developer must internalise this
- End-to-end tests must verify event flow through RabbitMQ, not just HTTP response codes
- Debugging cross-context workflows requires distributed tracing (OpenTelemetry + Jaeger), not just reading one log file
- The domain event schema in `packages/shared-events/` is a **public contract** between contexts — breaking changes require new versioned event types and backward-compatible consumers

### Neutral

- The domain event schema replaces HTTP API contracts as the integration point between bounded contexts. It must be versioned with the same discipline as an OpenAPI spec. A breaking change must introduce a new event type (e.g. `task.claimed.v2`) rather than modifying the existing one, so that old and new consumers can coexist during rolling migrations. This is standard DDD event versioning practice.

## DDD Glossary (as applied in this ADR)

| Term | Definition in RuralGig |
|---|---|
| **Bounded Context** | A self-contained part of the domain with its own model, language, and data ownership. Maps 1:1 to a NestJS service. |
| **Aggregate Root** | The single entry point for all changes to a cluster of domain objects. `Task`, `Worker`, `Wallet` are aggregate roots. Enforces all invariants. Raises domain events. |
| **Domain Event** | An immutable, past-tense record of something significant that happened in the domain. Named as business facts: `TaskClaimed`, not `UpdateTask`. |
| **Ubiquitous Language** | The shared vocabulary within a bounded context. `Task` in the Task Context is not the same concept as `PayoutInstruction` in the Payment Context. |
| **Anti-Corruption Layer (ACL)** | A translation layer in each subscribing context that converts incoming events from other contexts' language into the subscribing context's own domain model. Prevents language pollution. |
| **Event Storming** | A DDD collaborative workshop technique that maps domain events, commands, aggregates, and context boundaries on a timeline. Used to discover the bounded contexts in this system. |
| **Context Map** | A diagram showing how bounded contexts relate. In RuralGig, all contexts communicate via domain events over RabbitMQ — a Published Language pattern. |
| **Outbox Pattern** | A reliability pattern that ensures aggregate state changes and domain event publication are always atomic, preventing domain state and event log from diverging. |

## Related Decisions

- **ADR-002** — Why CQRS for the task feed. The task feed read model is built by projecting `TaskCreatedEvent` and `TaskClaimedEvent` into a Redis cache. This is only possible because domain events are the authoritative record of what happened.
- **ADR-003** — Why the Saga pattern over 2-phase commit for the payment workflow. The Saga is a DDD-native pattern for long-running cross-context transactions: a sequence of local transactions coordinated via domain events, with compensating transactions for rollback. It is the correct DDD answer to distributed consistency.
- **ADR-005** — Why each service owns its own PostgreSQL schema. This is the database-level enforcement of DDD bounded context autonomy. A shared database is incompatible with bounded contexts — it creates shared data ownership, which is the antithesis of the bounded context model. Schema-per-service and event-driven communication are two sides of the same DDD coin.
